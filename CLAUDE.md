# research-notes — operating manual

One markdown note per engaged-with paper (papers/<citekey>.md); topic maps in
topics/. Facts only in paper notes — no project-relevance judgments (those live in
project repos). Never store PDFs. Bump `updated` on every edit. After adding or
re-depthing a note, run `node scripts/build-catalog.js`.

`catalog.json` and `INDEX.md` are generated by `scripts/build-catalog.js` from
`papers/*.md` frontmatter — never hand-edit either file; run the script instead.

## INDEX.md format
One line per paper, grouped by topic, sorted by year desc:
`- [citekey](papers/<citekey>.md) — title | year | depth`
Read INDEX.md first; open paper notes only as needed.

## Workflows

### add <citekey or DOI/arXiv URL> [topic]
Create papers/<citekey>.md from the template at `depth: metadata`, filling
frontmatter from the identifier (fetch metadata if a URL/DOI is given). If an
abstract is available, fill Contribution and Key claims from it and set
`depth: abstract`. Placement only happens if a topic is named explicitly (sets
`topics:` and runs the `topic <t>` workflow); otherwise the note is born
`topics: []` — see specs/sync-placement.md. Run `node scripts/build-catalog.js`.

### deepen <citekey> [sections...]
Upgrade a note one rung: read only the named sections (or the minimum sections needed
— never the full PDF end-to-end), update Method/Evidence/Sections read, bump depth.
State what the upgrade changed.

### topic <name>
Create-or-update, idempotent: if `topics/<name>.md` doesn't exist, create it from the
template; either way, re-derive Shape of the field / Clusters / Reading queue strictly
from papers whose `topics:` frontmatter names `<name>` (never adds a paper the
frontmatter doesn't name). Preserve human-written Open questions verbatim and leave
Outputs untouched. Bump `updated` only if something changed. Run
`node scripts/build-catalog.js`. See specs/topic.md.

### propose [topic]
Analyze the whole repo (never opening paper-note bodies or topic-map prose) and write
a reviewable proposal to `scratch/propose-<date>.md`: placements, new clusters/topics,
restructuring, or defer — one action per candidate from `node scripts/propose-scan.js`.
Refuses to run if `doctor.js` reports any error finding. Decides nothing; a human
checks items to accept. See specs/propose.md.

### propose apply <report>
Executes exactly the checked (`- [x]`) items in a propose report: sets `topics:`
frontmatter for the named citekeys, runs the `topic` workflow once per affected map,
then build-catalog + doctor. A malformed checked line hard-stops before any write.
See specs/propose.md.

### survey <topic>
Draft a survey article from the topic map and its paper notes, citing by citekey,
flagging every claim that rests on a `depth: metadata` note as needing verification.
Output to a scratch file for review — never publish directly.

### link <citekey>
Print the full GitHub URL for the note (for pasting into a project repo), plus the
note's one-line Contribution as suggested link text.

### sync [batch size]
Run `node scripts/zotero-fetch.js`, then author notes for the queued items per
specs/sync.md and specs/sync-backlog.md (batches of ~15). Notes are born
`topics: []`, always — sync never opens or edits `topics/*.md`; placement is
`propose`'s job. Never write to Zotero; never invent citekeys. End with
build-catalog + doctor, then report the repo's unassigned-paper count with a
hint to run `propose`.

### doctor [--fix]
Run `node scripts/doctor.js` and explain findings. `--fix` regenerates generated
files only; never edit notes or topic maps to "fix" a finding — propose diffs.

### dedup
Turn doctor's D13 duplicate findings into a reviewable choice checklist. Reads
`doctor --json` + `propose-scan --json` + `catalog.json` (never note bodies),
coalesces D13 pairs into connected-component groups, and writes one checkbox
resolution checklist per group to `scratch/dedup-<date>.md` — candidate facts,
the human's decisions (survivor citekey, body to preserve, delete/defer), and
the derived link consequences. Report only: decides nothing, resolves nothing,
deletes nothing. See specs/dedup.md.

## Conventions

- **Citekeys:** generated and pinned by Better BibTeX in Zotero (recommended format:
  `authYYYYfirstword`). The citekey is the universal join key across Zotero, `.bib`
  files, this repo, and project repos.
- **Cost ladder discipline:** default engagement is `abstract`. `sections` requires a
  reason (recorded in the note); `full` should be rare enough to be an event.
- **Project-repo side (for reference, not part of this repo's setup):** each project
  keeps its own short survey/context notes and links here by URL; when a project
  discovers a factual error or a new connection, the fix is made *here*, once.
